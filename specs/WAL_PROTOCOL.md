# WAL-PROTOCOL.md — How to maintain Write-Ahead Log

Этот протокол определяет, как и когда AI должен обновлять `WAL.md`, чтобы обеспечить бесшовную передачу состояния между сессиями.

## 1. Когда обновлять WAL
- **В конце каждой сессии (обязательно)** — зафиксируй все изменения, выполненные за сессию.
- **Перед деструктивными операциями** — рефакторинг, удаление кода, изменение архитектуры. Сделай checkpoint в WAL.
- **При переключении контекста** — если человек перенаправляет тебя на другую задачу, сначала обнови WAL для текущей.

## 2. Формат записей
- Все ссылки на спецификации должны использовать полный URI:  
  `spec://<reverse-dns-модуля>/<документ>#<секция>`
- **Completed**: переноси реализованные пункты из `In Progress` в `Completed`. Можно добавлять краткий комментарий (например, «all tests pass»).
- **In Progress**:
  - `DONE`: уже реализованные подзадачи текущей фазы.
  - `TODO`: предстоящие шаги (с ссылками на спеки).
- **Known Issues**: указывай файл и строку, если возможно.
- **Decisions Pending**: формулируй вопрос чётко, предлагай варианты.
- **Session Context**: обновляй перед завершением сессии, чтобы следующая знала, с чего начать, какие файлы ключевые, и что **не трогать** (анти-инструкции).

## 3. Коллапсирование
Когда все подпункты секции завершены, можно заменить несколько строк одной

Детали реализации остаются в истории Git, не загромождай WAL.

## 4. Ограничение по размеру
Старайся держать WAL в пределах **3000 токенов**. Если файл разрастается — уплотняй завершённые разделы и выноси неактуальные детали.

## 5. Согласованность
Никогда не оставляй WAL в противоречивом состоянии. Если сессия прервалась аварийно, человек восстановит данные вручную, но твоя задача — минимизировать такие ситуации.
